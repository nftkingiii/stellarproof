# StellarProof

StellarProof is a hackathon build for verifiable off-chain invoice underwriting on Stellar.

A business can keep invoice-level data private, run the underwriting policy off-chain, and publish a small public result that is backed by a RISC Zero proof. The Stellar side is a Soroban attestor contract that calls the RISC Zero verifier-router interface before storing an attestation.

## What is built

- One-page web app and proof lab for the product flow.
- RISC Zero guest and host crates that generate a Groth16 proof on Linux.
- GitHub Actions proof worker that runs the real prover and publishes `app/proofs/latest-proof.json`.
- Soroban attestor contract that stores only verified eligible journals.
- Stellar testnet verifier router and StellarProof attestor deployment documented in `STELLAR_TESTNET.md`.
- Stellar testnet deploy script for the attestor contract.
- Railway-ready Node static server.

## Why this fits Stellar Hacks: Real-World ZK

- Real-world use case: invoice factoring and working-capital underwriting.
- ZK is load-bearing: the proof certifies that the risk score and eligibility decision came from the committed underwriting program.
- Stellar is load-bearing: the attestor contract is written for Soroban and verifies through the RISC Zero verifier-router interface before writing the public attestation.
- The app separates fast UX from heavy proof generation: Railway serves the interface, while GitHub Actions performs the Linux RISC Zero proving job.

## How the flow works

1. A business enters private invoice data.
2. The RISC Zero guest computes:
   - total invoice value
   - weighted risk score
   - eligible / not eligible
   - private input commitment
3. The public journal contains only:
   - applicant account
   - invoice count
   - total invoice value
   - weighted score
   - eligibility result
   - input commitment
4. The host writes `proof.txt` with `seal`, `image_id`, `journal_digest`, and `journal_bytes`.
5. GitHub Actions publishes the latest proof metadata to `app/proofs/latest-proof.json`.
6. The Soroban attestor initializes with the verifier router address and RISC Zero image ID, then calls `verify(seal, image_id, journal_digest)` before storing the journal.

## Run locally

Install Node.js, then run:

```bash
npm install
npm test
npm start
```

Open:

```text
http://127.0.0.1:3000/
```

The browser proof lab is intentionally instant. It simulates the product interaction and shows the connected Linux proof artifact. The cryptographic proof itself is generated by the GitHub Actions Linux workflow.

## Generate the real proof

Use the GitHub workflow:

```text
.github/workflows/stellarproof-linux.yml
```

The workflow:

1. installs Rust and RISC Zero tooling on Ubuntu,
2. runs the Node demo test,
3. tests the Soroban attestor contract,
4. builds the attestor Wasm,
5. runs `cargo run -p stellarproof-host --release`,
6. uploads `proof.txt`, and
7. commits `app/proofs/latest-proof.json` for the deployed app to read.

The latest checked-in proof JSON currently includes:

- source: `github-actions-risc-zero-groth16`
- image ID
- journal digest
- proof text SHA-256
- GitHub Actions run URL

## Deploy the Soroban attestor to Stellar testnet

Prerequisites:

- Stellar CLI installed and available as `stellar`.
- Rust target `wasm32v1-none` installed.
- A funded Stellar testnet identity configured in Stellar CLI.
- A RISC Zero verifier-router contract ID on the same network.

Deploy and initialize:

```powershell
./scripts/deploy-attestor-testnet.ps1 `
  -Source <stellar_cli_identity> `
  -RouterId <risc_zero_verifier_router_contract_id>
```

The script reads the latest RISC Zero image ID from `app/proofs/latest-proof.json`, builds the Wasm if needed, deploys `stellarproof_attestor`, calls `init(router, image_id)`, and writes `deployments/testnet.json`.

Submit a proof after deployment:

```powershell
./scripts/submit-attestation.ps1 `
  -Network testnet `
  -Source <stellar_cli_identity> `
  -AttestorId <attestor_contract_id> `
  -JournalXdr <public_journal_xdr> `
  -ProofFile proof.txt
```

